# Pre-Review Audit: Clinical NLP Entity Extraction

Senior ML engineering audit, 2026-06-10. Scope: `clinical_nlp/` (Python package), `src/lib/clinical-extraction/` (TS browser pipeline), `server.mjs`, tests, and docs. Every claim below was verified by reading the code and, where marked, by executing it. New tests documenting gaps use `pytest.mark.xfail(strict=True)` / vitest `it.fails`, so both suites stay green today and break loudly when a gap is fixed.

**Verdict in one paragraph.** This is an unusually well-structured prototype — typed schemas, a real PRF evaluation module, sentence-scoped NegEx-style negation in Python, abbreviation disambiguation, and honest "not production NLP" disclaimers. But it is two parallel pipelines that disagree with each other, the evaluation that exists has never been pointed at gold labels, the hybrid LLM path corrupted every span it returned, and three of the assertion classes the schema advertises (historical, family-history, conditional) are silently mapped to "present". Measured with the new harness, the default (NLP-mode) Python extractor has **0.26 recall overall and 0.00 recall for problems, symptoms, medications, and procedures** — it only knows vitals, two scores, pain, and SI.

---

## Blockers

**B1. Hybrid LLM mentions carried offsets into the wrong document — FIXED.**
`HybridExtractor` sends `compress_note_for_llm()` output to the LLM, then merged the returned mentions — whose offsets index the *compressed excerpt* — directly into mentions whose offsets index the *original note* (`extractors.py`). Verified empirically: "breast cancer" at offset 68 in the source sits at offset 53 in the excerpt. Every downstream consumer of spans (highlighting, dedupe, evaluation matching) would silently mis-align. Fix: `remap_mentions_to_source()` locates each excerpt line in the source, applies the delta, and validates that the remapped span reproduces the mention text exactly; unmappable mentions are dropped, not emitted corrupted. Covered by `tests/test_clinical_edge_cases.py::test_hybrid_llm_mention_offsets_map_back_to_source_text`. TODO left in code: drop-count telemetry; duplicated sentences (copy-forward) map to first occurrence.

**B2. LLM provider failure threw away all deterministic results — FIXED.**
`HybridExtractor.extract()` let `LLMExtractionError` propagate, so a provider outage on an "ambiguous" note returned *nothing* — including the vitals and SI mentions the regex layer had already found. Now degrades to NLP-only output (test: `test_hybrid_degrades_to_nlp_mentions_when_llm_fails`). TODO in code: return a result envelope with `escalation_failed=True` so degraded notes are routed to review instead of passing silently — this is currently the one remaining silent-failure path in the hybrid mode.

**B3. PHI exfiltration path: `Access-Control-Allow-Origin: *` on session endpoints — FIXED.**
`server.mjs` set wildcard CORS on `/api/*` while `/api/sessions/:id/export/:type` returns `sourceText` (the raw note) and session ids default to guessable `session-${Date.now()}`. Any web page open in a clinician's browser could enumerate and read sessions cross-origin. CORS is now opt-in via `CORS_ALLOW_ORIGIN`. TODO in code: authn and unguessable ids before real notes touch these endpoints; the in-memory `sessions` Map is also unbounded (memory-growth DoS).

**B4. No gold-standard evaluation loop — SCAFFOLDED.**
The Python `evaluation.py` module is genuinely good (exact + partial span matching, per-entity-type PRF, assertion-attribute accuracy) — and nothing in the repo ever ran it against labeled data. The TS "Eval Lab" is circular: `evaluationFixtures.ts` contains 20 synthetic notes whose `expectedCanonicalNames` were written from the same dictionaries the extractor matches against, and the unit test then asserts recall ≥ 0.9 with zero misses. That's a regression test for the dictionary, not an evaluation; it has no spans, no assertion gold, no entity-type gold, and can never reveal a miss on language the dictionary doesn't already contain. Scaffolded: `scripts/run_eval.py` (per-type PRF + assertion accuracy + `--min-f1` CI gate, validates gold span/text agreement on load) and `data/gold/seed_notes.jsonl` (6 synthetic notes covering negation lists, historical, family history, conditionals, abbreviations). First run output is the most informative artifact in this audit: overall partial F1 0.414, recall 0.261, and per-type recall 0.0 for problem/symptom/medication/procedure. Labeling plan below.

**B5. Python pipeline has no terminology normalization — SCAFFOLDED.**
`ClinicalMention.normalized_text` is a raw string; nothing maps to SNOMED CT, RxNorm, LOINC, or ICD-10-CM. Raw strings cannot populate FHIR CodeableConcepts, join to formularies or problem lists, or feed quality measures — this blocks interoperability outright. Scaffolded `clinical_nlp/normalization.py`: `ConceptCoding` (FHIR-style system URIs), a `TerminologyResolver` protocol, a `StaticTerminologyResolver` covering the deterministic concepts, and `normalize_mentions()` which flags uncoded mentions via `is_coded=False` rather than emitting them silently (tests in `tests/test_normalization.py`). The TS side is ahead — `terminologyMappings.ts` has ~1,100 lines of candidate codings — but note `TERMINOLOGY_SYSTEM_VERSIONS`: every vocabulary except ICD-10-CM FY2026 is "release not pinned", and a code without a release version is not safely comparable across time. TODO in module: real terminology service adapter (UMLS/RxNorm API/FHIR `$lookup`), release pinning, no logging of lookup strings.

**B6. TS dedupe makes "absent" sticky across contradicting mentions — FAILING TEST WRITTEN.**
`dedupeEntities.ts` merges all mentions of a canonical name into one entity and resolves assertion as: if *any* mention is absent, the entity is absent. Verified: `"Denies chest pain.\nNow reports chest pain worse with exertion."` produces one `chest pain` entity asserted **absent**. An actively symptomatic complaint reported as denied is the most clinically dangerous behavior found in this audit, and copy-forwarded notes make contradicting mentions the common case, not the corner case. Documented in `clinicalEdgeCases.test.ts` (`it.fails`). Proposed fix: keep assertion at the *mention* level (as the Python schema already does), and roll up to entity level with explicit conflict status (`conflicting` → forced review) instead of a sticky-absent rule.

**B7. PHI leaked into exception messages on the LLM path — FIXED.**
`validate_mention_plausibility()` embedded `mention.text` (raw clinical text) in raised `ValueError`s, which chain into `LLMExtractionError` and from there into whatever logs/observability wrap the caller. Errors now identify mentions by index, entity type, and span. Residual: validator messages still include numeric values (e.g. an implausible BP), which is acceptable; and `correction_context` echoes validation errors back to the same LLM provider that already holds the note — no *additional* disclosure, but see Q3 below on whether that provider should hold the note at all.

## Concerns

**C1. Two divergent implementations of the same clinical logic.** Negation, evaluation, abbreviation handling, and plausibility validation each exist twice (Python and TS) and already disagree: Python negation is sentence-scoped with termination cues and a hypothetical class; TS `negationRules.ts` is a fixed 36-character lookbehind with no termination cues and no hypothetical class. Empirically confirmed TS failures (all in `clinicalEdgeCases.test.ts`): `"No fever but cough is present."` → cough **absent** (no termination cue); `"No fever, chills, cough, vomiting, diarrhea, or headache."` → headache **present** (escapes the 36-char window — a false-positive *and* false-negative generator in the same rule); `"Rule out asthma."` → **present**. Whichever stack survives (Q1), port the Python resolver semantics.

**C2. Historical, family-history, and conditional assertions are schema fiction in both stacks.** `AssertionStatus` advertises HISTORICAL and FAMILY_HISTORY; no trigger ever assigns them in Python — `"History of myocardial infarction"`, `"Status post CABG"`, `"Mother has breast cancer"`, `"If chest pain recurs"`, and `"Patient educated on warning signs of stroke"` all resolve **PRESENT** (verified; now strict-xfail tests). TS handles family history only via the `FHx:` section header or hardcoded dictionary entries like "family history of breast cancer", and temporality only via a `PMH:` header — inline narrative is ignored (verified: `"History of asthma."` → present, no temporality). The hybrid TriagePolicy routes such text to the LLM, but in the default NLP mode (`OrchestratorConfig.mode = NLP`) these are silently wrong with confidence 0.92. Also: TriagePolicy matches ambiguous terms by raw substring — "mother" matches "smothered".

**C3. Confidence is not calibrated anywhere, and empty denominators report perfection.** TS confidence is a string heuristic (`term.length <= 3 ? "medium" : "high"`); Python regex concepts carry a constant 0.92. None of it is calibrated against outcomes, yet `TriagePolicy.min_confidence_for_skip = 0.75` makes routing decisions on it. Separately, `evaluation.safe_ratio()` returns 1.0 when the denominator is 0 — visible in the harness output, where four entity types with *zero predictions* report precision 1.000. Reasonable as a convention, misleading in a dashboard; report `None`/`n/a` instead.

**C4. The system never abstains; the HITL hook exists but gates nothing.** Both pipelines always emit. The good news: `uncertainty.ts` (`reviewPriority: high / needs-review / routine`) plus the reviewer-state controls in `EntityDetailPanel.tsx` are a real human-in-the-loop attachment point. The gap: nothing prevents exporting (JSON/FHIR/clipboard) entities that are `high` priority or unreviewed. Recommended: export gate requiring review acknowledgment for high-risk types (risk/SI, medications, allergies) and any `conflicting` assertion from B6's fix.

**C5. LLM drift would be invisible.** `LLMExtractionClient` has no model identifier; the extraction prompt is an unversioned inline string in `llm_bridge.py`; mentions don't record which model/prompt produced them; and there is no golden-set regression for the LLM path (the Pipeline Lab rows literally say "Pending structured LLM runs"). If the provider silently updates the model, nothing in this repo would notice. Becomes a blocker the day hybrid mode is enabled — see Q3 plan.

**C6. Reproducibility is half-done.** JS is properly locked (`package-lock.json`, 231 resolved packages). Python has no lockfile at all — `pyproject.toml` ranges only (`pydantic>=2.8,<3`, `ruff>=0.6`). Concretely observed: running the repo's own `ruff check` with a current ruff yields 141 findings on files the author presumably lints clean — the toolchain itself drifts. Add `uv lock`/`requirements.lock` and pin dev tools exactly. There is also no CI config in the repo; `run_tests.sh` is opt-in.

**C7. PHI at rest in the browser, and on the clipboard.** Full note text persists unencrypted in `localStorage` and IndexedDB (`sessionPersistence.ts`) and flows to `navigator.clipboard` from `DocumentOutputPanel`. Fine while inputs are synthetic; on shared clinical workstations this is cached PHI surviving logout. Before real notes: encrypt-at-rest or stop persisting source text, and add a session TTL. (Server-side: B3. No clinical text is written to server logs — verified the only `console.*` calls log paths and ports.)

**C8. Laterality and lexical robustness.** `detectLaterality()` matches `\b(l|left)\b` against the mention plus the *first 24 characters of the sentence* — verified: `"Patient left the clinic with knee pain."` lateralizes knee pain as **left**. Wrong-side documentation is a patient-safety class of error; require an anatomical anchor ("left knee", "L shoulder") rather than a bare verb match. There is also no misspelling tolerance anywhere (`"diabetis"`, `"hypertention"` → zero entities, verified) and no copy-forward/duplicate-paragraph detection — combined with B6, copy-forward actively corrupts assertions rather than just inflating counts.

**C9. Risk API silently zero-fills missing features.** `sanitizeRiskFeatures()` in `server.mjs` coerces absent or non-numeric features to 0 with no warning in the response, shifting the prediction rather than rejecting the request. The artifact's `__missing` indicator features mitigate this only for the two features that have them. Reject requests missing required features, or echo a `coercedFeatures` list.

## Nitpicks

`makeTermRegex` computes `isSymbolTerm` and then uses identical boundary strings in both branches — dead code. `attachDisambiguation` matches a resolution to an entity if *any* mention text equals the abbreviation, which can attach the wrong disambiguation when two entities share shorthand. `buildAmbiguousReviewEntities` only anchors the *first* occurrence of an ambiguous abbreviation. `dist/` build output is committed. `merge_mentions` dedupes only exact `(type, start, end)` keys, so an LLM span one character off from a deterministic span yields a near-duplicate pair. Python `NLPExtractor`'s SI regex (`\bSI\b`) will fire inside e.g. French/Spanish quoted text; confidence 0.86 with no context check on a suicide-risk label deserves more care than most labels, not less.

## Evaluation harness and labeling plan

The harness (`scripts/run_eval.py`) runs the configured orchestrator over `data/gold/*.jsonl`, validates gold span/text agreement, and prints overall + per-entity-type precision/recall/F1 plus assertion accuracy on matched mentions, with a `--min-f1` exit-code gate for CI. Seed corpus: 6 synthetic notes. To make it real:

1. **Corpus**: 200–300 notes, stratified across primary care, mental health, PT/SLP, and mixed, including the ugly cases this audit exercised: denial lists, copy-forward bloat, inline history/FHx, conditionals, misspellings, ambiguous abbreviations. Source from synthetic generation plus properly de-identified real notes (Safe Harbor or Expert Determination) — gold files live in the repo, so nothing identifiable, ever.
2. **Schema**: annotate spans with `entity_type` (the existing `EntityType` enum) and `assertion` (the existing `AssertionStatus` enum — guidelines adapted from the i2b2/2010 assertion task). The JSONL format in the harness docstring is the contract.
3. **Process**: written annotation guideline with worked examples → dual annotation on a 50-note calibration round → adjudication → measure inter-annotator agreement (target: span-level F1 ≥ 0.85 between annotators; Cohen's κ ≥ 0.7 on assertion). Iterate guidelines until met, then annotate the remainder single-pass with 15% dual-annotated overlap for drift monitoring. Tooling: Label Studio or INCEpTION export → JSONL converter.
4. **Splits and gating**: freeze a held-out test split on day one; dev split drives dictionary/rule work; CI runs `run_eval.py --min-f1 <current - 0.02>` on dev so regressions block merge; test split is touched only for releases.

## Three questions for the author, turned into a plan

**Q1 — "Which pipeline is the product?"** Python `clinical_nlp` and TS `clinical-extraction` implement extraction, negation, evaluation, and validation twice, and they already disagree (C1). Maintaining both means every clinical-logic fix lands twice or diverges.
*Plan:* Decide within a sprint. Recommended: Python becomes the single extraction engine served behind `/api/sessions/:id/extract` (the endpoint already stubs this); the TS pipeline is demoted to UI rendering of server responses, and its negation/eval modules are frozen with a deprecation note. Port the Python resolver's termination-cue and hypothetical semantics before cutover; the TS edge-case tests in this audit become the acceptance tests.

**Q2 — "What's the operating point: review-gated or autonomous, and for which consumers?"** Default-PRESENT on unknown assertion, always-emit behavior, and uncalibrated 0.92 confidence are all fine if every entity passes a human reviewer, and all unacceptable if output feeds billing, problem lists, or risk models directly.
*Plan:* Write the answer down as per-entity-type policy: target precision/recall per type, confidence threshold below which the system abstains (emits a review task, not an entity), and which types (risk/SI, medications, allergies) are *never* exportable unreviewed. Implement the export gate on `reviewPriority`/reviewer-state (C4) and the `conflicting` assertion status (B6). Calibrate confidence against the gold corpus from B4 so the thresholds mean something.

**Q3 — "What is the actual LLM deployment story — provider, BAA, model pinning, redaction?"** The hybrid path ships raw note excerpts to an external provider with no BAA gating, no model version, no prompt version, and no drift detection (C5, B7).
*Plan:* Before enabling hybrid mode: (1) extend `LLMExtractionClient` with `model_id` and pin it — reject "latest" aliases; (2) version the prompt (hash + semver in repo) and stamp `model_id`/`prompt_version` into `ClinicalMention.attributes` for every LLM-derived mention; (3) stand up a nightly golden-set run of the LLM path against the frozen test split with alerting on per-type F1 deltas > 2 points — this is the drift tripwire; (4) document the data path and confirm BAA/zero-retention terms with the provider, or route through an in-VPC model; (5) keep B2's degradation envelope so provider incidents surface in review queues instead of silently shrinking recall.
